# 项目设计

## 1. 选题判断

“AI 投资管家”很适合作为 Agent 求职项目，因为它天然包含多源数据、工具调用、长期状态、检索、规划、定时任务和高风险人工确认。但应避免把目标写成“预测涨跌、自动赚钱”，这既难以可信评测，也容易让项目沦为提示词演示。

推荐项目名称：**FinAgent：可追溯的 AI 投资研究与资产监控助手**。

一句话介绍：系统围绕用户的模拟持仓主动收集数据、研究市场事件、生成带证据的风险报告，并在人工确认边界内提供可解释的调仓方案草稿。

## 2. 目标用户与核心场景

目标用户是持有股票、基金或黄金，但没有时间持续整理信息的个人投资者。

核心场景：

- 资产总览：统一查看模拟持仓、成本、现值、盈亏、资产类别和集中度。
- 每日简报：回答“今天发生了什么，与我的持仓有什么关系？”
- 事件研究：围绕一个国际或行业事件，形成“事实 → 影响路径 → 受影响资产 → 风险与反例”的证据链。
- 风险监控：价格波动、单一资产集中度、行业暴露或重大事件满足规则时触发提醒。
- 方案草稿：基于用户预先定义的约束，给出多种调仓情景和理由，但不执行交易。
- 研究记忆：记住用户偏好、历史研究结论和反馈，并允许查看、修改、删除。

## 3. 与普通聊天机器人的差异

项目必须呈现以下工程证据：

- **数据可追溯**：每条市场事实带来源、抓取时间和有效时间。
- **计算可复现**：盈亏、权重和风险指标由 Python 函数计算并有单元测试。
- **行为可控制**：工具有输入输出 Schema、超时、重试、权限和审计记录。
- **状态可恢复**：长任务中断后能够从检查点继续。
- **结论可评测**：有固定问题集、检索指标、工具调用成功率和历史回放。
- **风险可管理**：提示不确定性；涉及调仓时必须人工确认；永不自动交易。

## 4. 功能边界

### 首个可用版本包含

- 模拟持仓录入；
- 一种云端模型；
- 资产计算工具与模拟行情工具；
- CLI 对话；
- 结构化回答、日志、错误处理和测试。

### 最终版本包含

- 云端/本地模型路由；
- 股票、基金、黄金、新闻和宏观数据适配器；
- Web API 与简单前端；
- 记忆、RAG、MCP、可靠工作流、多智能体研究；
- 定时简报与提醒；
- 评测、可观测性、Docker 和 CI。

### 暂不包含

- 真实资金托管和券商下单；
- 保证收益、个股涨跌预测；
- 高频或分钟级交易；
- 未经许可抓取付费内容；
- 第一版就训练或微调大模型。

## 5. 简历价值

项目完成后，应能用数据写成类似下面的经历，而不是只罗列技术：

> 设计并实现模块化投资研究 Agent，支持多模型路由、带引用的混合检索、持久化记忆与 MCP 工具接入；构建历史事件回放评测集，对检索命中率、工具调用成功率、回答忠实度、延迟与成本进行持续评估；通过检查点和人工确认机制提高长任务可靠性与高风险操作安全性。

其中的数字必须来自真实评测结果，项目后期再填写，不能编造。

## 6. 验收标准

最终演示至少包含：

1. 导入一组匿名模拟持仓；
2. 询问某一宏观事件对持仓的影响；
3. 展示 Agent 的计划、工具调用、证据引用和结构化报告；
4. 切换云端模型与本地模型并比较质量、延迟和成本；
5. 中断并恢复一次研究任务；
6. 通过 MCP 接入至少一个自建金融工具服务；
7. 展示自动化测试、评测报告、追踪记录和 GitHub Actions；
8. 展示一次需要用户批准的调仓方案草稿。
