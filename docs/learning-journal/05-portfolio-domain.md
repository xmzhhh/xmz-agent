## 核心原理（用自己的话）

领域模型不是简单的数据容器，而是业务规则的第一道边界。例如 `Quote` 不只是价格，
还必须回答“什么资产、什么币种、截至什么时间、来自哪里、是否延迟”。缺少这些信息，
模型即使给出数值，也无法证明它是当前可用的行情。

投资组合计算器是一段纯函数式业务逻辑：它不联网、不调用 LLM、不保存数据库。同一组
持仓和行情一定得到同一快照，因此可以可靠单元测试，也可以在未来历史回放中复现。

## 数据流

```text
外部持仓/行情字典
  → Pydantic 校验和规范化
  → Holding / Quote
  → 币种、重复代码和行情完整性检查
  → Decimal 估值计算
  → ValuedHolding 明细
  → PortfolioSnapshot 汇总
```

## 关键公式

```text
持仓成本 = 持有数量 × 单位平均成本
当前市值 = 持有数量 × 当前价格
浮动盈亏 = 当前市值 - 持仓成本
收益率 = 浮动盈亏 ÷ 持仓成本 × 100%
持仓权重 = 单项市值 ÷ 组合总市值 × 100%
HHI = Σ(持仓百分比权重²)
```

使用百分比计算 HHI 时，其范围为 0～10000。只有一项资产时 HHI 为 10000；两项等权
资产约为 5000；持仓越分散，HHI 越低。它只能描述市值集中度，不直接等于投资风险。

## 为什么金额使用 Decimal

`float` 使用二进制表示小数，很多十进制金额无法精确保存。例如 `0.1 + 0.2` 可能不会
精确等于 `0.3`。本阶段在金融模型边界直接拒绝 float，只接受：

- 十进制字符串，例如 `"4.80"`；
- 整数，例如 `1000`；
- 已经正确构造的 `Decimal`。

金额和展示百分比统一使用 `ROUND_HALF_UP` 保留两位，避免不同模块选择不同舍入方式。

## 关键设计与取舍

### 1. 明细先舍入，再计算汇总

每项持仓的成本和市值先保留到分，组合总额再对展示明细求和。这样用户看到的明细之和
一定等于总额，不会出现界面上相差一分钱的问题。

### 2. 权重必须严格等于 100.00%

每项权重独立保留两位后，三项等权资产会得到 `33.33% × 3 = 99.99%`。计算器把舍入
差额调整给市值最大的持仓，最终得到 `33.34%、33.33%、33.33%`。修正最大仓位可以
避免微小仓位在极端情况下被调整成负数。

### 3. 组合时间取最旧行情

如果黄金行情是 10:00，基金行情是 09:30，整个组合只能声称“截至 09:30”有效。使用
最新时间会掩盖基金数据较旧的事实。

### 4. 第一版不做汇率换算

人民币成本不能直接减去美元市值。本阶段遇到币种不一致就抛出
`CurrencyMismatchError`，汇率 Provider 将在后续独立实现。

### 5. 空组合返回零值快照

用户刚开始使用时可能尚未录入持仓。返回一个结构稳定的空快照比抛出除零异常更方便
CLI 和未来 API 展示；空组合收益率使用 `None`，因为它在数学上没有定义。

## 遇到的问题

### 问题 1：如何防止 Decimal 接收已经失真的 float

- **问题现象**：Pydantic 默认可以把 float 转成 Decimal，但此时二进制误差可能已经产生。
- **产生原因**：数据类型转换只能改变表示形式，不能恢复 float 在上一步丢失的十进制精度。
- **排查思路**：把校验位置前移到 Pydantic 转换之前，区分字符串、整数、Decimal 和 float。
- **解决方法**：使用 `BeforeValidator` 定义 `DecimalInput`，显式拒绝 float 和 bool。
- **学到的知识**：仅仅“最终字段类型是 Decimal”还不够，还要控制数据到达 Decimal 前的来源。

另外通过 `allow_inf_nan=False` 拒绝 `NaN` 和无穷大；它们虽然属于 Decimal 特殊值，
但无法形成合法的价格、金额或比较结果。

### 问题 2：独立舍入后的权重和不是 100%

- **问题现象**：三项等权持仓各自显示 33.33%，合计只有 99.99%。
- **产生原因**：无限循环小数必须舍入，每个位置独立舍入会累积显示误差。
- **排查思路**：比较原始权重、展示权重及其合计，并考虑把差额分配给哪个位置最安全。
- **解决方法**：计算差额 `100.00 - 已舍入权重和`，把差额调整到最大市值持仓。
- **学到的知识**：金融系统不仅要保证公式正确，还要保证用户看到的明细与汇总一致。

### 问题 3：不同市场时区的行情如何比较

- **问题现象**：无时区 `datetime` 无法说明 10:00 属于北京时间还是 UTC，也不能可靠比较。
- **产生原因**：naive datetime 缺少 UTC 偏移信息。
- **排查思路**：构造 `09:30+08:00` 与 `02:00+00:00` 的行情，验证它们按真实时刻比较。
- **解决方法**：`Quote` 强制 `as_of` 包含时区；Python 会把 aware datetime 转换到统一时间线比较。
- **学到的知识**：行情时间必须同时保存本地时间和时区语义，只有日期字符串不够。

### 问题 4：新增文件首次需要统一格式化

- **问题现象**：Ruff 首次检查前有两个新文件需要自动格式化。
- **产生原因**：长类型声明和测试参数在初稿后尚未经过项目统一排版。
- **排查思路**：先运行 Ruff format，再分别执行 lint、mypy 和 pytest。
- **解决方法**：自动格式化后四项质量门禁全部通过。
- **学到的知识**：格式、静态类型和运行行为属于不同质量维度，不能互相替代。
