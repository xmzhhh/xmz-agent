# 学习日记 09：模拟持仓管理与资产面板

日期：2026-07-22

计划版本：v0.2.2

相关 Issue / PR：Issue #13 / PR 待创建

## 本阶段目标

把已有的持仓模型、真实行情路由和投资组合计算器连接成一个可操作的资产面板。当前阶段使用
异步内存仓库保存模拟持仓，通过 FastAPI 提供 API，通过 Jinja2、原生 JavaScript 和 CSS 提供
网页界面；本阶段不接数据库、不执行真实交易，也不把资产面板接入 Agent 工具。

## 已确定的核心边界

- `017811` 作为支持持仓的公募基金，通过 AKShare 查询每日确认净值。
- `JD-ZS-GOLD` 作为支持持仓的京东金融浙商银行积存金，必须使用用户手工录入的京东卖出价估值。
- `XAU-CNY-GRAM` 只作为 GoldAPI 国际黄金参考价，不能替代京东卖出价，也不能录入为持仓。
- 投资组合计算器负责毛市值、毛收益、预计卖出费、预计到账金额和净收益等确定性公式。
- 仓位权重与 HHI 继续使用毛市值计算，避免手续费改变风险集中度口径。
- 本阶段的数据只保存在进程内存中，程序重启后持仓和手工价格都会清空。
- 每个阶段只使用一个功能分支，完成代码、测试和文档后再创建唯一 PR，不再建立收尾分支。

## 遇到的问题

### 问题 1：GitHub 连接器没有权限创建 Issue

- **问题现象**：尝试通过 GitHub 连接器创建 Phase 6 Issue 时返回 `403 Resource not accessible by integration`。
- **产生原因**：当前连接器身份对该仓库没有创建 Issue 所需的写权限；这是外部权限边界，不是 Issue
  标题或正文格式错误。
- **排查思路**：根据 HTTP 403 和错误中的 integration 提示，先区分“请求内容错误”和“授权不足”；
  随后确认本机已经登录 GitHub CLI，可以在用户授权范围内操作该公开仓库。
- **解决方法**：改用已登录的 `gh` CLI 创建 Issue，并在创建后读取 Issue 内容进行复核。
- **学到的知识**：不同 GitHub 工具使用不同的身份和权限。某个连接器没有写权限，并不代表仓库或
  本机 GitHub CLI 也没有权限；遇到 403 应先检查授权边界。

### 问题 2：PowerShell 把未加引号的 `@me` 当成特殊语法

- **问题现象**：第一次调用 `gh issue create --assignee @me` 时，GitHub CLI 报告
  `flag needs an argument: --assignee`。
- **产生原因**：`@me` 在到达 `gh` 之前被 PowerShell 按自身语法处理，导致 `--assignee` 实际没有
  收到参数。
- **排查思路**：根据报错确认失败发生在命令参数解析阶段，而不是 GitHub 用户不存在；随后检查
  shell 对 `@` 开头文本的处理差异。
- **解决方法**：把参数写成带引号的 `'@me'`，让它作为完整字符串传给 `gh`。
- **学到的知识**：同一条 CLI 命令会先经过 shell 解析，再由目标程序解析。在 Windows 上编写自动化
  命令时，要区分 PowerShell、CMD 和 Git CMD 的转义规则。

### 问题 3：Issue 标题正常，但中文正文全部变成问号

- **问题现象**：Issue #13 创建成功，中文标题显示正常，但正文中的大部分中文都被替换成 `?`。
- **产生原因**：中文正文通过 PowerShell 管道传给 `gh` 时发生编码转换，字符在到达 GitHub 之前已经
  丢失；浏览器只是在展示已经损坏的正文。
- **排查思路**：标题能够正常显示，说明 GitHub 页面和仓库支持中文；正文稳定变成问号，说明问题在
  正文传输链路。直接读取远程 Issue 后也能看到问号，排除了浏览器字体或页面渲染问题。
- **解决方法**：使用 UTF-8 Markdown 临时文件保存完整正文，再通过
  `gh issue edit 13 --body-file <文件>` 更新；随后读取远程 Issue 并在网页复核，确认中文恢复。
- **学到的知识**：Windows 下向 GitHub 提交较长中文 Issue 或 PR 正文时，应优先使用 UTF-8
  `--body-file`，避免依赖 PowerShell 管道的默认编码；写入后还要验证远端实际内容。

### 问题 4：局部补丁完成后 Ruff 报告四个 Python 文件需要格式化

- **问题现象**：费率计算专项测试已经全部通过，但 `ruff format --check` 报告本轮修改的模型、
  计算器和两个测试文件需要重新格式化。
- **产生原因**：新增的长函数调用和字段声明超过项目格式规则允许的单行长度，同时 Windows 工作区
  的既有行尾与补丁写入行尾需要由格式器统一。
- **排查思路**：先单独运行 Ruff lint、format check 和 mypy，确认问题属于代码排版而不是公式、
  类型或测试断言；再核对待格式化文件全部属于本轮修改范围。
- **解决方法**：只对这四个本轮文件执行 `ruff format`，随后重新执行 Ruff lint、format check、
  mypy 和 pytest。
- **学到的知识**：测试通过只说明运行行为符合断言，不代表格式和静态类型门禁已经通过；局部补丁后
  应分别执行测试、格式、lint 和类型检查，并把自动格式化限制在任务相关文件。

### 问题 5：新增目录和仓库后 Ruff 要求格式化三个既有文件

- **问题现象**：第一次并行执行目录/仓库专项测试、Ruff lint 和 format check 时，format check
  报告 `portfolio/__init__.py`、`errors.py` 和 `models.py` 需要重新格式化。
- **产生原因**：三个既有文件都被本轮补丁新增了导出项、异常类或输入模型，局部写入后的行尾和空行
  需要与项目格式规则统一；新建的 `catalog.py`、`repository.py` 和测试文件本身无需调整。
- **排查思路**：根据 Ruff 明确列出的文件限制处理范围，确认没有要求格式化任务外文件；由于并行
  命令在 format check 失败时只返回了该失败结果，格式化后再分别执行专项测试、lint 和 mypy。
- **解决方法**：只对 Ruff 点名的三个文件运行 `ruff format`，随后专项 52 项测试、Ruff 和 mypy
  均通过。
- **学到的知识**：并行质量检查可以缩短等待时间，但某一任务失败时输出聚合可能只展示首个失败；
  修复后应拆开复验，不能据此假设其他并行任务已经成功或失败。

### 问题 6：Pydantic 运行时接受字符串，但 mypy 拒绝直接构造 Decimal 字段

- **问题现象**：手工价格与 Dashboard Service 的 18 项专项测试全部通过，但 mypy 报告 22 个
  `arg-type` 错误，指出 `HoldingCreate(quantity="100")` 和 `ManualPriceInput(price="850")`
  把字符串传给了静态类型为 `Decimal` 的字段。
- **产生原因**：Pydantic 在运行时会把 API 中的数字字符串转换为 `Decimal`，但 mypy 只分析 Python
  构造函数签名，不会模拟 Pydantic 的运行时转换，因此直接构造模型时要求传入真正的 `Decimal`。
- **排查思路**：测试通过说明运行时校验没有问题；错误全部集中在直接模型构造处，说明不是计算公式
  或仓库协议错误。对比项目已有代码后，确认外部字符串输入应该通过 `model_validate()` 进入模型。
- **解决方法**：匿名演示常量和测试辅助函数统一改为
  `Model.model_validate({"price": "850"})`，明确模拟 JSON/表单字符串输入；无效 float 测试也通过
  `model_validate()` 触发运行时边界校验。修改后 pytest、Ruff 和 mypy 全部通过。
- **学到的知识**：Pydantic 的“运行时可转换输入”和 Python 的“静态构造参数类型”是两个层次。
  外部不可信字典使用 `model_validate()`，内部已经是精确类型的数据可以直接构造模型。

## 第一小阶段：费率感知的投资组合估值

### 目标

在现有毛市值和毛收益计算基础上，增加每项持仓的预计卖出费率、预计卖出费、预计到账金额、
净盈亏和净收益率。所有计算仍由纯 Python `PortfolioCalculator` 完成，不访问网络或仓库。

### 计算口径

```text
预计卖出费 = 毛市值 × 预计卖出费率 ÷ 100
预计到账金额 = 毛市值 - 预计卖出费
净盈亏 = 预计到账金额 - 持仓成本
净收益率 = 净盈亏 ÷ 持仓成本 × 100%
```

费率字段使用“百分数”语义：输入 `0.5` 表示 `0.5%`。金额按 `ROUND_HALF_UP` 保留两位小数；
组合手续费和净金额由已经舍入的单项明细相加，保证明细与汇总完全一致。

## 第二小阶段：资产目录与异步内存持仓仓库

### 资产目录为什么独立存在

用户只应该输入资产代码、数量、持仓均价和预计卖出费率。资产名称、类型、币种以及价格来源属于
系统维护的产品知识，如果也允许用户填写，同一个 `017811` 可能被伪造成不同名称或错误类型。
因此 `AssetCatalog` 作为白名单保存规范元数据：

- `017811`：允许持仓，使用行情 Provider 估值。
- `JD-ZS-GOLD`：允许持仓，使用手工京东卖出价估值。
- `XAU-CNY-GRAM`：只显示国际黄金参考价，不允许创建持仓。

### 仓库为什么现在就使用异步接口

FastAPI 的请求处理和未来 SQLite 仓库都会使用异步边界。本阶段虽然只是内存字典，仍通过
`HoldingRepository` 协议定义异步 CRUD，让下一阶段的 Dashboard Service 不依赖具体存储方式。
`InMemoryHoldingRepository` 使用 `asyncio.Lock` 把“检查后写入”保护成一个原子操作，避免并发请求
同时创建相同代码。

### 演示组合怎样保证原子性

`load_demo` 只允许在空仓库执行。它先在锁内校验全部目录条目和重复代码，只有整批数据都合法后
才一次性写入字典；任何一项失败，仓库仍然为空。是否处于 Fake 模式和具体匿名数据由下一阶段的
Dashboard Service 决定，仓库不读取配置。

## 第三小阶段：手工黄金卖价与 Dashboard Service

### 手工价格为什么单独建仓库

京东积存金没有可直接调用的公开成交价接口，因此用户需要手工录入当前卖出价。手工价格与持仓的
生命周期不同：修改持仓数量不应修改价格，但删除 `JD-ZS-GOLD` 持仓时必须同步清除价格。
`InMemoryManualPriceRepository` 只保存价格记录；服务端在写入时补全代码、币种和 `recorded_at`，
用户无法伪造录入时间。

### 900 秒边界怎样处理

Dashboard Service 生成快照时计算 `当前服务端时间 - recorded_at`：

- 恰好 900 秒仍然有效。
- 严格超过 900 秒抛出 `ManualPriceStaleError`。
- 价格缺失抛出 `ManualPriceNotFoundError`。
- 录入时间晚于当前时间也拒绝使用，避免系统时钟回拨产生“永不过期”价格。

缺失或过期会在访问基金、GoldAPI 之前失败，既防止残缺组合，也避免浪费真实 API 额度。

### 必要行情与可选参考价怎样分开

`017811` 基金净值和京东手工卖出价属于组合估值的必要数据，任何一项失败都不生成快照。
`XAU-CNY-GRAM` 只在存在京东黄金持仓时单独查询；GoldAPI 抛出已知市场数据异常后，组合快照仍然
成功，参考栏状态改为 `unavailable`。空仓和纯基金组合的参考栏为 `not_requested`，不会查询黄金。

### 服务层为什么还需要一把状态锁

持仓仓库和手工价格仓库各自只能保证内部操作原子。Dashboard Service 增加状态锁，保护“删除黄金
持仓并清理价格”和“载入两项演示持仓并写入演示价格”等跨仓库操作。生成面板时只在锁内复制
内存快照，网络请求前立即释放锁，避免慢行情阻塞持仓 CRUD。
