## 本阶段目标

把已有的持仓模型、真实行情路由和投资组合计算器连接成一个可操作的资产面板。当前阶段使用
异步内存仓库保存模拟持仓，通过 FastAPI 提供 API，通过 Jinja2、原生 JavaScript 和 CSS 提供
网页界面；本阶段不接数据库、不执行真实交易，也不把资产面板接入 Agent 工具。

## 已确定的核心边界

- `017811` 作为支持持仓的公募基金，通过 AKShare 查询每日确认净值。
- `JD-ZS-GOLD` 作为支持持仓的京东金融浙商银行积存金，必须使用用户手工录入的京东卖出价估值。
- `XAU-CNY-GRAM` 只作为 GoldAPI 国际黄金参考价，不能替代京东卖出价，也不能录入为持仓。
- 投资组合计算器负责毛市值、毛收益、预计卖出费、预计到账金额和净收益等确定性公式。
- 仓位权重与 HHI 继续使用毛市值计算，避免手续费改变风险集中度口径。
- 本阶段的数据只保存在进程内存中，程序重启后持仓和手工价格都会清空。
- 每个阶段只使用一个功能分支，完成代码、测试和文档后再创建唯一 PR，不再建立收尾分支。

## 遇到的问题

### 问题 1：GitHub 连接器没有权限创建 Issue

- **问题现象**：尝试通过 GitHub 连接器创建 Phase 6 Issue 时返回 `403 Resource not accessible by integration`。
- **产生原因**：当前连接器身份对该仓库没有创建 Issue 所需的写权限；这是外部权限边界，不是 Issue
  标题或正文格式错误。
- **排查思路**：根据 HTTP 403 和错误中的 integration 提示，先区分“请求内容错误”和“授权不足”；
  随后确认本机已经登录 GitHub CLI，可以在用户授权范围内操作该公开仓库。
- **解决方法**：改用已登录的 `gh` CLI 创建 Issue，并在创建后读取 Issue 内容进行复核。
- **学到的知识**：不同 GitHub 工具使用不同的身份和权限。某个连接器没有写权限，并不代表仓库或
  本机 GitHub CLI 也没有权限；遇到 403 应先检查授权边界。

### 问题 2：PowerShell 把未加引号的 `@me` 当成特殊语法

- **问题现象**：第一次调用 `gh issue create --assignee @me` 时，GitHub CLI 报告
  `flag needs an argument: --assignee`。
- **产生原因**：`@me` 在到达 `gh` 之前被 PowerShell 按自身语法处理，导致 `--assignee` 实际没有
  收到参数。
- **排查思路**：根据报错确认失败发生在命令参数解析阶段，而不是 GitHub 用户不存在；随后检查
  shell 对 `@` 开头文本的处理差异。
- **解决方法**：把参数写成带引号的 `'@me'`，让它作为完整字符串传给 `gh`。
- **学到的知识**：同一条 CLI 命令会先经过 shell 解析，再由目标程序解析。在 Windows 上编写自动化
  命令时，要区分 PowerShell、CMD 和 Git CMD 的转义规则。

### 问题 3：Issue 标题正常，但中文正文全部变成问号

- **问题现象**：Issue #13 创建成功，中文标题显示正常，但正文中的大部分中文都被替换成 `?`。
- **产生原因**：中文正文通过 PowerShell 管道传给 `gh` 时发生编码转换，字符在到达 GitHub 之前已经
  丢失；浏览器只是在展示已经损坏的正文。
- **排查思路**：标题能够正常显示，说明 GitHub 页面和仓库支持中文；正文稳定变成问号，说明问题在
  正文传输链路。直接读取远程 Issue 后也能看到问号，排除了浏览器字体或页面渲染问题。
- **解决方法**：使用 UTF-8 Markdown 临时文件保存完整正文，再通过
  `gh issue edit 13 --body-file <文件>` 更新；随后读取远程 Issue 并在网页复核，确认中文恢复。
- **学到的知识**：Windows 下向 GitHub 提交较长中文 Issue 或 PR 正文时，应优先使用 UTF-8
  `--body-file`，避免依赖 PowerShell 管道的默认编码；写入后还要验证远端实际内容。

### 问题 4：局部补丁完成后 Ruff 报告四个 Python 文件需要格式化

- **问题现象**：费率计算专项测试已经全部通过，但 `ruff format --check` 报告本轮修改的模型、
  计算器和两个测试文件需要重新格式化。
- **产生原因**：新增的长函数调用和字段声明超过项目格式规则允许的单行长度，同时 Windows 工作区
  的既有行尾与补丁写入行尾需要由格式器统一。
- **排查思路**：先单独运行 Ruff lint、format check 和 mypy，确认问题属于代码排版而不是公式、
  类型或测试断言；再核对待格式化文件全部属于本轮修改范围。
- **解决方法**：只对这四个本轮文件执行 `ruff format`，随后重新执行 Ruff lint、format check、
  mypy 和 pytest。
- **学到的知识**：测试通过只说明运行行为符合断言，不代表格式和静态类型门禁已经通过；局部补丁后
  应分别执行测试、格式、lint 和类型检查，并把自动格式化限制在任务相关文件。

### 问题 5：新增目录和仓库后 Ruff 要求格式化三个既有文件

- **问题现象**：第一次并行执行目录/仓库专项测试、Ruff lint 和 format check 时，format check
  报告 `portfolio/__init__.py`、`errors.py` 和 `models.py` 需要重新格式化。
- **产生原因**：三个既有文件都被本轮补丁新增了导出项、异常类或输入模型，局部写入后的行尾和空行
  需要与项目格式规则统一；新建的 `catalog.py`、`repository.py` 和测试文件本身无需调整。
- **排查思路**：根据 Ruff 明确列出的文件限制处理范围，确认没有要求格式化任务外文件；由于并行
  命令在 format check 失败时只返回了该失败结果，格式化后再分别执行专项测试、lint 和 mypy。
- **解决方法**：只对 Ruff 点名的三个文件运行 `ruff format`，随后专项 52 项测试、Ruff 和 mypy
  均通过。
- **学到的知识**：并行质量检查可以缩短等待时间，但某一任务失败时输出聚合可能只展示首个失败；
  修复后应拆开复验，不能据此假设其他并行任务已经成功或失败。

### 问题 6：Pydantic 运行时接受字符串，但 mypy 拒绝直接构造 Decimal 字段

- **问题现象**：手工价格与 Dashboard Service 的 18 项专项测试全部通过，但 mypy 报告 22 个
  `arg-type` 错误，指出 `HoldingCreate(quantity="100")` 和 `ManualPriceInput(price="850")`
  把字符串传给了静态类型为 `Decimal` 的字段。
- **产生原因**：Pydantic 在运行时会把 API 中的数字字符串转换为 `Decimal`，但 mypy 只分析 Python
  构造函数签名，不会模拟 Pydantic 的运行时转换，因此直接构造模型时要求传入真正的 `Decimal`。
- **排查思路**：测试通过说明运行时校验没有问题；错误全部集中在直接模型构造处，说明不是计算公式
  或仓库协议错误。对比项目已有代码后，确认外部字符串输入应该通过 `model_validate()` 进入模型。
- **解决方法**：匿名演示常量和测试辅助函数统一改为
  `Model.model_validate({"price": "850"})`，明确模拟 JSON/表单字符串输入；无效 float 测试也通过
  `model_validate()` 触发运行时边界校验。修改后 pytest、Ruff 和 mypy 全部通过。
- **学到的知识**：Pydantic 的“运行时可转换输入”和 Python 的“静态构造参数类型”是两个层次。
  外部不可信字典使用 `model_validate()`，内部已经是精确类型的数据可以直接构造模型。

### 问题 7：uv 全局缓存目录在当前进程中被拒绝访问

- **问题现象**：执行 `uv lock --check` 时，uv 无法读取用户目录下的全局缓存
  `C:\Users\xmz\AppData\Local\uv\cache`，并报告访问被拒绝。
- **产生原因**：当前自动化进程对该全局缓存目录的权限与用户交互式终端不同；依赖声明和
  `uv.lock` 本身没有冲突。
- **排查思路**：先区分“依赖解析失败”和“缓存目录不可写”，再使用项目内的新缓存目录执行同一条
  只读锁文件检查。如果新缓存能够完成解析，说明问题不在依赖版本。
- **解决方法**：临时设置 `UV_CACHE_DIR` 为项目已忽略的 `.codex_tmp/uv-cache` 后重新执行
  `uv lock --check`，72 个包解析成功。
- **学到的知识**：包管理器报错不一定来自依赖；缓存、临时目录和文件锁同样会影响命令。排查时应
  尽量只替换缓存位置，不删除全局缓存或改动依赖声明。

### 问题 8：FastAPI TestClient 出现 Starlette 弃用警告

- **问题现象**：最初的 35 项 API/配置/CLI 专项测试全部通过，但 Starlette 提示当前
  `TestClient` 与 httpx 的兼容方式已弃用，建议迁移。
- **产生原因**：测试使用 FastAPI 的同步 `TestClient`，它触发了当前 Starlette 版本中的兼容层；
  这不影响断言结果，但会让未来依赖升级存在风险。
- **排查思路**：确认警告来自测试客户端而非应用路由，再检查项目已经直接依赖 httpx，因此无需为了
  测试再引入新的 HTTP 客户端依赖。
- **解决方法**：测试改用 `httpx.AsyncClient` 和 `ASGITransport`，直接在内存中请求 FastAPI；同时
  显式进入 `app.router.lifespan_context(app)`，保留启动/关闭生命周期验证。修改后 35 项专项测试
  无警告通过。
- **学到的知识**：HTTP 端点测试不必启动真实端口。ASGI Transport 可以测试完整 HTTP 序列化和
  状态码；但它不会自动管理应用 lifespan，测试必须显式进入生命周期上下文。

### 问题 9：pytest 复用旧临时目录时报告 WinError 5

- **问题现象**：API 测试中 34 项通过、1 项在准备 `tmp_path` 时失败，错误是旧的
  `.codex_tmp/phase6_web_tmp` 无法删除，同时 pytest 缓存目录也不可写。
- **产生原因**：Windows 上旧临时目录仍带有当前进程无法处理的权限或占用状态；失败发生在 fixture
  准备阶段，不是被测配置、API 或业务逻辑报错。
- **排查思路**：根据栈追踪中的 `shutil.rmtree` 和 `tmp_path` 定位到测试基础设施，再观察同一轮 API
  测试已经全部通过，排除业务回归。
- **解决方法**：每轮测试在系统临时目录下使用带随机 GUID 的全新 `--basetemp`，并以
  `-p no:cacheprovider` 禁用非必要 pytest 缓存；随后专项 35 项和全量 234 项全部通过。
- **学到的知识**：测试失败要先看“失败发生在哪一层”。fixture 建立失败不能算业务测试失败；使用
  唯一临时目录比强行删除权限不明的旧目录更安全。

### 问题 10：mypy 发现过时抑制和分支具体类型冲突

- **问题现象**：运行测试和 Ruff 均通过，但 mypy 报告 `Settings()` 上的 `type: ignore` 已不再需要，
  并认为同一个 `provider` 变量不能先后接收 Fake 与 Routing 两种具体 Provider。
- **产生原因**：`LLM_API_KEY` 改为可选后，旧的构造器抑制已经过时；另一个错误来自 mypy 根据 Fake
  分支首次赋值，把变量推断成了 `FakeMarketDataProvider`，因此拒绝 Real 分支的路由 Provider。
- **排查思路**：第一个错误按 `unused-ignore` 判断为应删除的历史补丁；第二个错误比较两个实现的共同
  接口，确认它们都满足 `MarketDataProvider` 协议。
- **解决方法**：删除无用 `type: ignore`，并在分支前把变量声明为
  `provider: MarketDataProvider`。复验后 73 个 Python 文件通过 strict mypy。
- **学到的知识**：类型忽略不是永久答案，行为变化后要及时删除；依赖倒置不仅是架构概念，也能通过
  协议类型准确表达“上层只关心共同能力”。

### 问题 12：浏览器验收工具不支持 networkidle 等待方式

- **问题现象**：本地页面已经能打开，但第一次要求浏览器等待 `networkidle` 时，控制工具报告当前
  实现不支持该等待状态，因而没有返回 DOM 快照。
- **产生原因**：这是浏览器控制工具支持的 Playwright 子集限制，不是页面网络请求一直无法结束。
- **排查思路**：确认导航本身已经完成，再改为等待标准 `domcontentloaded`，随后读取可访问性 DOM；
  页面显示 Fake 模式已连接、空仓快照和完整表单，证明前端脚本已经执行。
- **解决方法**：使用 `domcontentloaded` 作为初始页面条件；交互后不使用固定休眠，而是等待“2 项持仓”
  或具体成功提示等确定 DOM 状态。
- **学到的知识**：端到端测试应等待业务可观察结果，而不是过度依赖某个浏览器实现的全局加载状态。
  对异步页面来说，“目标文本出现”通常比“网络完全空闲”更接近真实验收目标。

### 问题 13：本地验收命令超时后 Uvicorn 进程仍在监听端口

- **问题现象**：浏览器验收结束后，原本承载 Uvicorn 的长时间命令已经超时，但检查发现 8765 端口仍由
  子进程监听。
- **产生原因**：命令执行器结束了等待，不等于它启动的服务器子进程一定同步退出；如果不复核，会在
  后续运行中产生端口占用。
- **排查思路**：浏览器标签清理后单独查询 8765 端口的监听状态和拥有进程，确认确实存在本轮启动的
  Uvicorn，而不是根据命令超时状态猜测它已经停止。
- **解决方法**：只终止本轮明确记录的服务器进程，再次查询端口并确认 `PORT_8765_CLOSED`。
- **学到的知识**：启动临时服务的自动化验收必须成对包含“启动确认”和“关闭确认”；命令结束、浏览器
  关闭和服务退出是三个不同状态。

### 问题 14：wheel 资源检查把目录条目误算成网页文件

- **问题现象**：`uv build` 成功，输出也列出了 HTML、CSS 和 JavaScript，但第一版检查仍报告
  “wheel 中缺少网页资源”。
- **产生原因**：`tar -tf` 同时返回 `static/`、`templates/` 两个目录和三个真实文件；断言把全部五个
  匹配项与预期的三个文件比较，造成假失败。
- **排查思路**：先阅读构建输出，确认三个目标文件都存在，再检查验收脚本的筛选正则和计数对象，发现
  它没有排除目录。
- **解决方法**：筛选条件改为只接受以 `.html`、`.css` 或 `.js` 结尾的包内路径；重新构建后准确得到
  三个文件并输出 `WHEEL_WEB_ASSETS_OK`。
- **学到的知识**：验收脚本本身也需要调试。遇到“命令输出证明成功、最终断言却失败”时，应核对断言
  的统计口径，不能为了让脚本变绿而修改已经正确的打包配置。

## 第一小阶段：费率感知的投资组合估值

### 目标

在现有毛市值和毛收益计算基础上，增加每项持仓的预计卖出费率、预计卖出费、预计到账金额、
净盈亏和净收益率。所有计算仍由纯 Python `PortfolioCalculator` 完成，不访问网络或仓库。

### 计算口径

```text
预计卖出费 = 毛市值 × 预计卖出费率 ÷ 100
预计到账金额 = 毛市值 - 预计卖出费
净盈亏 = 预计到账金额 - 持仓成本
净收益率 = 净盈亏 ÷ 持仓成本 × 100%
```

费率字段使用“百分数”语义：输入 `0.5` 表示 `0.5%`。金额按 `ROUND_HALF_UP` 保留两位小数；
组合手续费和净金额由已经舍入的单项明细相加，保证明细与汇总完全一致。

## 第二小阶段：资产目录与异步内存持仓仓库

### 资产目录为什么独立存在

用户只应该输入资产代码、数量、持仓均价和预计卖出费率。资产名称、类型、币种以及价格来源属于
系统维护的产品知识，如果也允许用户填写，同一个 `017811` 可能被伪造成不同名称或错误类型。
因此 `AssetCatalog` 作为白名单保存规范元数据：

- `017811`：允许持仓，使用行情 Provider 估值。
- `JD-ZS-GOLD`：允许持仓，使用手工京东卖出价估值。
- `XAU-CNY-GRAM`：只显示国际黄金参考价，不允许创建持仓。

### 仓库为什么现在就使用异步接口

FastAPI 的请求处理和未来 SQLite 仓库都会使用异步边界。本阶段虽然只是内存字典，仍通过
`HoldingRepository` 协议定义异步 CRUD，让下一阶段的 Dashboard Service 不依赖具体存储方式。
`InMemoryHoldingRepository` 使用 `asyncio.Lock` 把“检查后写入”保护成一个原子操作，避免并发请求
同时创建相同代码。

### 演示组合怎样保证原子性

`load_demo` 只允许在空仓库执行。它先在锁内校验全部目录条目和重复代码，只有整批数据都合法后
才一次性写入字典；任何一项失败，仓库仍然为空。是否处于 Fake 模式和具体匿名数据由下一阶段的
Dashboard Service 决定，仓库不读取配置。

## 第三小阶段：手工黄金卖价与 Dashboard Service

### 手工价格为什么单独建仓库

京东积存金没有可直接调用的公开成交价接口，因此用户需要手工录入当前卖出价。手工价格与持仓的
生命周期不同：修改持仓数量不应修改价格，但删除 `JD-ZS-GOLD` 持仓时必须同步清除价格。
`InMemoryManualPriceRepository` 只保存价格记录；服务端在写入时补全代码、币种和 `recorded_at`，
用户无法伪造录入时间。

### 900 秒边界怎样处理

Dashboard Service 生成快照时计算 `当前服务端时间 - recorded_at`：

- 恰好 900 秒仍然有效。
- 严格超过 900 秒抛出 `ManualPriceStaleError`。
- 价格缺失抛出 `ManualPriceNotFoundError`。
- 录入时间晚于当前时间也拒绝使用，避免系统时钟回拨产生“永不过期”价格。

缺失或过期会在访问基金、GoldAPI 之前失败，既防止残缺组合，也避免浪费真实 API 额度。

### 必要行情与可选参考价怎样分开

`017811` 基金净值和京东手工卖出价属于组合估值的必要数据，任何一项失败都不生成快照。
`XAU-CNY-GRAM` 只在存在京东黄金持仓时单独查询；GoldAPI 抛出已知市场数据异常后，组合快照仍然
成功，参考栏状态改为 `unavailable`。空仓和纯基金组合的参考栏为 `not_requested`，不会查询黄金。

### 服务层为什么还需要一把状态锁

持仓仓库和手工价格仓库各自只能保证内部操作原子。Dashboard Service 增加状态锁，保护“删除黄金
持仓并清理价格”和“载入两项演示持仓并写入演示价格”等跨仓库操作。生成面板时只在锁内复制
内存快照，网络请求前立即释放锁，避免慢行情阻塞持仓 CRUD。

## 第四小阶段：配置边界、应用装配、FastAPI 与 CLI

### 为什么把 LLM_API_KEY 改为按需校验

资产面板的持仓 CRUD、Fake 行情和确定性计算都不调用大模型。如果 `Settings` 在加载时就要求
`LLM_API_KEY`，用户即使只想离线查看面板也无法启动。因此配置模型允许模型 Key 缺失，只有创建
`BailianModelProvider` 时才调用 `require_llm_api_key()`。这不是降低聊天功能的安全要求：
`finagent chat` 仍会在发起网络请求前明确失败，只是无关功能不再被模型配置耦合。

`MARKET_DATA_MODE=real` 不同，因为本阶段 Real 面板固定会装配 GoldAPI 国际黄金参考 Provider；为了
避免服务器启动后才发现认证缺失，Settings 会在 Real 模式下提前要求 `GOLDAPI_API_KEY`。

### 组合根怎样屏蔽 Fake 与 Real 的实现差异

`finagent.web.composition` 是本小阶段唯一知道全部具体实现的“组合根”：

```text
Fake 模式 → FakeMarketDataProvider ─┐
                                    ├→ MarketDataService
Real 模式 → RoutingMarketDataProvider┘
                                    ↓
内存持仓仓库 + 内存手工价仓库 → PortfolioDashboardService
```

Fake 模式使用固定基金净值和国际黄金参考价，允许载入匿名演示组合；Real 模式组合 AKShare 基金
Provider 与 GoldAPI Provider，并禁用演示组合。两种实现都通过 `MarketDataProvider` 协议交给
`MarketDataService`，所以 API 路由和 Dashboard Service 不需要判断数据源类型。

### FastAPI 层负责什么、不负责什么

`create_app()` 使用应用工厂创建独立内存状态，并注册 `/api/v1` 下的资产目录、持仓 CRUD、手工价格、
面板快照、演示组合和健康检查端点。HTTP 层只做四件事：

1. 用 Pydantic 把外部 JSON 校验为领域输入模型；
2. 调用 `PortfolioDashboardService`；
3. 用 response model 把 `Decimal` 金融数值稳定序列化为字符串；
4. 把领域异常映射为 404、409、422、503 或 504，并保持统一 `error.code/message` 结构。

金融公式、持仓规则、900 秒新鲜度和行情降级都没有复制到路由函数中。请求校验错误也不回显完整
请求正文，防止未来扩展字段后意外把敏感输入记录或返回。应用 lifespan 退出时调用 `service.close()`，
沿服务链关闭真实 HTTP Provider。

### CLI 为什么默认只监听 127.0.0.1

`finagent dashboard` 默认启动 `127.0.0.1:8000`，只有本机能访问。用户显式传入
`--host 0.0.0.0` 等非本机地址时，CLI 会提示当前面板没有登录认证、会向局域网开放读写接口。
这条提示不是身份认证的替代品；它只是在本阶段明确的可信局域网人工验收范围内降低误用风险。
端口参数在进入 Uvicorn 前限制为 1～65535，错误输入由 argparse 直接拒绝。

## 第五小阶段：Jinja2 与原生 JavaScript/CSS 资产面板

### 为什么页面只是一层 API 客户端

FastAPI 根路径 `/` 只用 Jinja2 返回页面结构，并通过 `/static` 提供 CSS 和 JavaScript。页面启动后读取
`/api/v1/health`、`/assets`、`/holdings` 和 `/dashboard`，所有毛市值、净到账、收益率、卖出费用、
权重与 HHI 都直接展示 API 结果。JavaScript 只做本地化金额/时间格式化和正负颜色判断，不复制任何
Python 金融公式，因此后端公式修正后所有界面会自动使用同一结果。

### 快照失败时为什么仍然显示持仓

持仓 CRUD 数据和组合估值快照是两个不同层次。比如用户新增 `JD-ZS-GOLD` 后尚未录入卖出价，
`GET /dashboard` 必须返回 409，防止生成残缺估值；但这条持仓本身已经合法写入仓库。前端因此先请求
`/holdings`，再单独尝试 `/dashboard`：快照失败时清空旧汇总并显示“等待估值”，但仍保留持仓行的
编辑和删除按钮，用户才能修复缺少的价格或删除错误持仓。

### 前端怎样避免把外部文本当成 HTML

基金名称来自白名单，但行情来源可能来自外部 Provider。动态表格全部使用 `textContent` 和
`document.createElement()` 写入，不使用 `innerHTML` 拼接数据；资产代码进入 URL 前使用
`encodeURIComponent()`。表单金额始终按字符串提交，编辑时也直接回填 API 的 Decimal 字符串，避免
先转换为 JavaScript 浮点数再把误差写回后端。

### 页面包含哪些操作和信息

- 新增、编辑和删除受支持持仓，资产代码在编辑模式下不可修改。
- 录入和清除京东积存金卖出价，展示服务端记录时间。
- Fake 模式空仓时原子载入匿名演示组合；Real 模式自动禁用按钮。
- 展示毛市值、预计到账、净盈亏、净收益率、预计卖出费和持仓成本。
- 展示每项持仓的毛/净口径、仓位、行情来源、行情时间和延迟标记。
- 展示最大仓位、HHI、资产类别权重以及可降级的国际黄金参考栏。
- 使用 1100、760 和 440 像素断点适配桌面、平板和手机；表格在自身容器内横向滚动，不让整个页面
  产生横向溢出。
