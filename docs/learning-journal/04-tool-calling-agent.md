## 本阶段目标

- 新增 Agent 与 CLI 离线测试，不访问网络、不消耗模型额度。

## 核心原理（用自己的话）

普通聊天只有一次模型调用：用户消息发给模型，模型直接返回文字。Agent 工具调用至少
包含两个模型步骤：第一次模型不回答最终结论，而是返回 `ToolCall`；应用执行对应的
Python 工具并形成 `role=tool` 消息；第二次把完整消息序列发给模型，模型读取可靠的工具
结果后组织最终自然语言回答。

模型只有“提出工具调用请求”的权力，真正的工具白名单、参数校验和执行权都在应用侧。
这正是 Function Calling 比让模型自由生成 Python 代码更可控的原因。

## 完整消息流

```text
system    定义 FinAgent 身份、边界和工具使用原则
user      提出“计算仓位比例”的问题
assistant 返回 ToolCall(name="calculate_position_ratio", arguments=...)
tool      返回带相同 tool_call_id 的结构化计算结果
assistant 读取工具结果后生成最终回答
```

这里的两个 `assistant` 不是重复消息：第一个是工具请求，第二个才是面向用户的最终回答。
`tool_call_id` 用来把工具结果与某一次调用精确对应；一次响应包含多个工具调用时尤其重要。

## 关键设计与取舍

### 1. Agent 与 CLI 分层

`ToolCallingAgent` 只处理模型、工具和历史；`cli.py` 只处理键盘输入和文本展示。未来加入
FastAPI 时可以直接复用 Agent，不需要把终端循环复制到 Web 层。

### 2. 事务式历史更新

一轮执行先在 `working_messages` 副本中推进。只有模型最终返回文本时，才整体替换正式
历史。如果中途网络失败、响应异常或超过最大步数，正式历史仍停留在上一轮完整状态。

### 3. 工具失败返回模型

参数错误和未知工具会生成 `ok=false` 的 JSON tool 消息，而不是立刻终止程序。模型可以
根据错误信息修正参数、换工具或解释失败。不过重试仍受 `max_steps` 限制。

### 4. 最大步数限制

默认一次提问最多调用模型 5 次。最后一次如果仍请求工具，会在执行工具前终止，因为
已经没有下一次模型调用能够消费结果；这也避免未来带副作用工具被无意义执行。

### 5. 工具暂时串行执行

一次模型响应可能包含多个 `ToolCall`。当前按返回顺序串行执行，流程清晰、结果稳定。
以后只有在工具相互独立、无副作用且并发收益明确时，才考虑 `asyncio.gather` 并行执行。

## 遇到的问题

### 问题 1：成功和失败工具结果缺少统一判别字段

- **问题现象**：上一阶段 `ToolResult` 只有 `tool_name` 和 `data`，而本阶段需要把工具
  失败也返回模型；两种 JSON 结构缺少统一判断方式。
- **产生原因**：上一阶段只关注成功执行，还没有进入 Agent 错误恢复流程。
- **排查思路**：从模型读取 tool 消息的角度比较成功与失败结构，确认它们都需要明确的
  `ok` 字段。
- **解决方法**：成功 `ToolResult` 增加固定的 `ok=true`；工具异常转换成包含 `ok=false`、
  `error_type` 和 `error` 的 JSON。
- **学到的知识**：模块接口会随真实调用场景逐步稳定；小阶段提前测试能用很低成本发现
  协议缺口。

### 问题 2：如何定义最大步数的停止位置

- **问题现象**：如果最后一次模型调用仍请求工具，是先执行工具再报超限，还是直接停止？
- **产生原因**：模型调用次数和工具执行次数不是同一个概念；工具未来还可能产生副作用。
- **排查思路**：检查最后一次工具结果是否还有机会被模型消费，并考虑重复执行写入型工具
  的风险。
- **解决方法**：`max_steps` 统计模型调用次数；最后一步仍请求工具时，在执行前抛出
  `AgentStepLimitError`。
- **学到的知识**：Agent 循环的终止条件不仅关系 token 成本，也关系工具副作用和状态
  一致性。

### 问题 3：新文件首次检查需要 Ruff 格式化

- **问题现象**：初稿中两个 Python 文件未符合 Ruff 的统一排版。
- **产生原因**：新增较长的测试和调用代码在初稿后尚未执行自动格式化。
- **排查思路**：先运行 `ruff format`，再运行 lint，区分机械格式与逻辑规则问题。
- **解决方法**：由 Ruff 自动格式化后重新执行 lint、mypy 和 pytest。
- **学到的知识**：格式化应在最终质量门禁前执行，但自动格式化通过不代表类型和行为正确。

### 真实模型验收证据

- **运行环境**：PyCharm，项目 `.venv` 解释器。
- **验收问题**：3000 元黄金持仓占 10000 元总资产的比例。
- **最终答案**：黄金持仓占比为 30.0%，剩余资产占比为 70%。
- **调用轨迹**：`['calculate_position_ratio']`。
- **结论**：答案、确定性公式和工具调用轨迹一致，证明“模型选择工具—应用执行—结果
  回传—模型总结”的真实百炼闭环已经跑通，而不是模型直接心算得到答案。

## 如何人工验收

在 PyCharm 中打开并运行 `scripts/step02_check_agent_tools.py`。预期输出应同时包含：

```text
最终回答：……30%……
实际工具调用：['calculate_position_ratio']
```

只有答案为 30% 不足以证明 Agent 工作，因为模型可能自己算出了结果；必须检查“实际
工具调用”列表中确实出现了 `calculate_position_ratio`。

也可以运行 `finagent chat`，输入涉及仓位计算或模拟行情的问题，观察最终回答。当前行情
是固定教学数据，回答必须明确说明不是实时数据。
