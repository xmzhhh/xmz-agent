# 学习日记 07：真实基金净值与国际黄金参考价

日期：2026-07-15 至 2026-07-17
版本：v0.2.0
相关 Issue / PR：Issue #7 / PR #8（已合并）

## 本阶段目标

在现有 ``MarketDataProvider``、``MarketDataService`` 和 ``Quote`` 契约之上，接入两个
真实但用途不同的数据源：使用 AKShare 查询场外公募基金的最新已确认净值，使用 GoldAPI
查询 XAU/CNY 国际黄金参考价。所有外部数据仍要经过统一校验，且不能把参考价、昨日净值
或估算值表述成用户能够立即成交的实时价格。

## 调研后确认的真实业务口径

### 京东金融中的浙商银行财富金积存

- 第一版由用户手工录入持有克数、持仓均价、京东买入价和京东卖出价。
- 当前页面实测买入手续费为 0，卖出手续费约为卖出金额的 0.4%。
- 京东页面的持有金额和持有收益均未扣除未来卖出手续费。
- GoldAPI 的 XAU/CNY 只能用于国际市场参考和价差比较，不能代替京东卖出价计算真实到账。

### 蚂蚁财富中的东方人工智能主题混合C（017811）

- 当前持仓来自一笔买入，第一版可先实现单批次输入，但模型应为未来多批次保留扩展边界。
- 持有金额等于持有份额乘以最新已确认单位净值，不是盘中实时成交价。
- 交易日 15 点前申请赎回，最终按当天基金公司确认的净值结算；提交时只能根据上一已知净值
  给出估算金额。
- 当前页面实测持有 7～29 天的赎回费率为 0.5%，实际费用以基金公司确认结果为准。

## 数据源选型

- **基金净值主源**：AKShare，第一只验收基金为 017811。
- **基金权威核对源**：东方基金官网。
- **国际黄金参考价**：[GoldAPI XAU/CNY 官方接口示例](https://www.goldapi.io/price/XAU/CNY/python)。
- **个人持仓与京东双向报价**：第一版手工输入，后续再研究本地截图识别。

## 核心原理（用自己的话）

### 1. 可选数据源不应该阻断其他功能

百炼密钥是当前 CLI Agent 启动的必需配置；GoldAPI 只是 Phase 4 新增的可选行情源。如果在
构造任何 ``Settings`` 时都强制 GoldAPI 密钥存在，那么没有使用黄金功能的聊天、基金测试
和离线单元测试也会失败。因此配置层允许 ``goldapi_api_key=None``，只有创建 GoldAPI
Provider 时才通过 ``require_goldapi_api_key`` 明确报错。

### 2. 缓存只能保存统一模型

``QuoteCache`` 保存已经校验的 ``Quote``，不保存 GoldAPI 原始 JSON 或 AKShare DataFrame。
这样缓存命中和真实请求返回的对象遵守同一契约，上层也不会因为缓存而依赖供应商字段。

### 3. TTL 使用单调时钟

缓存关心“过去了多少秒”，不关心当前日历时间。系统时间可能因为人工修改或网络校时发生
跳变，而 ``time.monotonic`` 只向前推进，更适合判断 15 分钟是否已经过去。测试通过注入
可控时钟推进时间，不需要真正等待。

### 4. 同步第三方库需要隔离在线程中

AKShare 当前使用同步 requests 请求。如果直接在 ``async def get_quote`` 中调用它，那么一次
网络等待会占住事件循环线程，模型流式输出、其他行情请求和超时调度都无法继续运行。
``asyncio.to_thread`` 把阻塞调用交给工作线程，Provider 对上层仍然保持异步接口。需要注意，
取消等待并不能强制杀死已经运行的工作线程，因此仍要依靠上游超时、缓存和并发限制控制资源。

### 5. Provider 是防止数据语义混淆的边界

AKShare 返回 pandas DataFrame，而组合计算器只认识 ``Quote``。Provider 在边界处完成基金代码、
列名、日期、单位净值、币种和数据时效语义的校验。特别是基金净值只有每日确认值，因此即使
它是当前能查询到的最新记录，也必须标记为延迟数据，不能称作盘中实时行情。

### 6. 金融价格必须同时表达计价币种和计量单位

GoldAPI 的 ``price`` 是每金衡盎司人民币价格，而积存金持仓数量使用克。1 金衡盎司约等于
31.1035 克，如果直接把 ``price`` 乘以持有克数，会产生约 31.1 倍的估值错误。因此 Provider
读取官方提供的 ``price_gram_24k``，并把单位编码进 ``XAU-CNY-GRAM`` 资产代码。现有 ``Quote``
只能表达币种、不能单独表达计量单位，后续领域模型升级时可以增加显式 ``price_unit`` 字段。

### 7. HTTP Provider 不应该悄悄自动重试

GoldAPI 免费套餐有月度请求额度，429 也可能表示额度已耗尽。如果 Provider 在内部自动重试，
一次用户操作可能消耗多次额度，而且上层无法解释实际请求次数。本阶段每次缓存未命中只请求
一次，并把超时、连接、鉴权和限流转换为不同异常；以后由 Service 根据异常类型执行可观察、
有次数上限的重试。

### 8. 人工验收脚本和可测试流程需要分层

``scripts/step05_check_real_market_data.py`` 只负责 PyCharm/命令行入口、参数、配置和退出码；真正的
双数据源检查流程放在 ``finagent.data.diagnostics``。这样 pytest 只导入安装在 ``src`` 下的
正式模块，不依赖项目根目录是否出现在 ``sys.path``，而人工脚本仍然可以直接运行。真实
Provider 通过默认值创建，测试则注入 Fake Provider，复用同一流程但绝不联网。

## 遇到的问题

### 问题 1：Git status 扫描 Codex 临时测试目录时提示权限不足

- **问题现象**：``git status --short`` 提示无法打开 ``.codex_tmp/pytest-of-xmz`` 和
  ``.codex_tmp/pytest_cache``。
- **产生原因**：项目根目录中的工具临时目录没有被 Git 忽略，其中部分子目录权限不允许
  当前 Git 进程遍历。
- **排查思路**：确认 ``.codex_tmp`` 没有被 Git 跟踪，并使用忽略规则阻止 Git 继续扫描。
- **解决方法**：在 ``.gitignore`` 中加入 ``.codex_tmp/``，随后 ``git status`` 不再警告。
- **学到的知识**：运行缓存和测试临时文件既不属于源码，也可能包含特殊权限，应该在创建后
  及时加入忽略规则。

### 问题 2：首次运行 uv add 无法访问用户级缓存

- **问题现象**：安装 AKShare 和 httpx 时，uv 无法打开
  ``C:\Users\xmz\AppData\Local\uv\cache``，返回 ``os error 5``。
- **产生原因**：当前受控执行环境默认只能写项目工作区，而 uv 的全局缓存位于工作区外。
- **排查思路**：错误发生在依赖解析前，路径明确指向用户缓存，因此不是包版本冲突或网络故障。
- **解决方法**：获得用户授权后允许 uv 访问其用户级缓存，再次运行安装成功；后续添加
  ``pandas-stubs`` 时遇到同一限制，也沿用相同的授权边界处理。
- **学到的知识**：依赖安装同时涉及项目文件、全局缓存和网络；权限问题要与解析失败、下载失败
  分开诊断。

### 问题 3：uv 修正第三方包的非法版本声明

- **问题现象**：安装时多次出现 ``Fixing invalid version specifier``，把 ``>=3.6,`` 修正为
  ``>=3.6``。
- **产生原因**：AKShare 某个传递依赖的包元数据包含尾随逗号，不符合当前版本规范。
- **排查思路**：继续观察最终解析、构建和安装结果，而不是看到 warning 就判断安装失败。
- **解决方法**：uv 自动规范化声明，最终成功安装 AKShare 1.18.64 及其依赖。
- **学到的知识**：warning 表示需要关注但不等于失败；应以退出码、锁文件和后续导入测试作为
  最终证据。

### 问题 5：pytest 无法写入系统临时目录和旧缓存目录

- **问题现象**：首轮 pytest 中 114 项通过，但使用 ``tmp_path`` 的配置测试在创建

- **产生原因**：受控执行环境不能写用户系统临时目录，仓库中旧 pytest 缓存也带有当前进程

- **排查思路**：根据 traceback 区分“fixture 建立失败”和“被测代码失败”；全部已执行测试均

### 问题 6：Git 提示 LF 将在工作区转换为 CRLF

- **问题现象**：``git diff`` 对若干文本文件提示下次写入时会把 LF 转换成 CRLF。
- **产生原因**：Windows Git 的行尾转换配置与仓库当前文件行尾不同。
- **排查思路**：``git diff --check`` 已通过，说明没有尾随空格或补丁格式错误；该信息属于
  工作区行尾提示，不是代码行为或测试失败。
- **当前处理**：本阶段不批量改写全仓库行尾，避免产生与功能无关的大面积 diff。
- **下一步**：后续生产化阶段再统一评估是否增加 ``.gitattributes`` 固定文本行尾策略。

### 问题 7：Ruff 发现测试文件存在未使用导入

- **问题现象**：首次检查 AKShare 测试时，Ruff 报告 ``datetime.timedelta`` 已导入但未使用。
- **产生原因**：编写测试过程中曾计划控制缓存时间，最终改为只测试缓存命中，但遗留了导入。
- **排查思路**：根据 Ruff 的 F401 位置确认该名称在文件中没有任何引用。
- **解决方法**：删除未使用导入，再次运行 Ruff 后通过。
- **学到的知识**：重构测试场景后也要清理导入；静态检查能发现不影响运行、但会降低可读性的
  残留代码。

### 问题 8：mypy 无法理解 pandas 类型和日期解析返回值

- **问题现象**：131 个测试全部通过，但 mypy 报告 pandas 缺少类型声明、AKShare 返回 ``Any``，
  并且 ``pd.to_datetime(object)`` 无法匹配严格重载。
- **产生原因**：pandas 运行包不包含 mypy 所需的完整声明；AKShare 也没有类型声明，DataFrame
  单元格在静态类型层面只能视为 ``object``。
- **排查思路**：先区分“运行行为正确”和“静态类型信息不足”；安装 pandas 官方生态的 stub，
  再只在无法从无类型第三方库推导的边界进行显式收窄。
- **解决方法**：把 ``pandas-stubs`` 加入开发依赖；用 ``cast(pd.DataFrame, ...)`` 描述 AKShare
  已知返回契约；日期单元格先转字符串，再交给 ``pd.to_datetime`` 的标量重载解析。
- **学到的知识**：类型声明包只服务开发期检查，不改变运行逻辑；``cast`` 也不做运行时转换，
  所以真实字段仍必须由 Provider 校验。

### 问题 9：requests 超时最初会被误判为响应解析错误

- **问题现象**：初版只把 ``OSError`` 转成连接异常，但 AKShare 底层 requests 的 ``Timeout``
  和 ``ConnectionError`` 不保证继承内置 ``OSError``，会落入通用异常分支。
- **产生原因**：只按 Python 内置 I/O 异常设计，没有检查第三方 HTTP 客户端的异常继承体系。
- **排查思路**：结合 AKShare 当前实现确认其使用 requests，再为 requests 超时建立专门离线测试。
- **解决方法**：将 requests 声明为直接依赖，并同时捕获 ``RequestException`` 与 ``OSError``，
  统一转换为 ``MarketDataConnectionError``。
- **学到的知识**：领域异常映射要依据实际客户端异常树；否则上层重试策略会因为错误分类而失效。

### 问题 10：GoldAPI 错误码文档的补充检索发生网络失败

- **问题现象**：官方页面已经确认端点、认证头和主要字段，但继续检索错误码说明时，搜索请求
  返回网络错误。
- **产生原因**：开发环境到搜索服务的临时网络连接失败，不是 GoldAPI Key 或项目代码错误。
- **排查思路**：先保留已经从 GoldAPI 官方页面确认的契约；错误处理不依赖不稳定的错误正文，
  而是依据标准 HTTP 状态语义设计，并通过 MockTransport 离线测试。
- **解决方法**：使用 401/403 表示鉴权、404 表示不存在、429 表示限流或额度、5xx 表示服务端
  响应失败；不把未经确认的供应商错误 JSON 字段写死进 Provider。
- **学到的知识**：外部文档暂时不可用时，应缩小到已经确认的最小契约，并用可替换的适配层
  隔离不确定部分，不能凭印象伪造响应结构。

### 问题 11：首次静态检查发现 UTC 和 AnyHttpUrl 类型写法不统一

- **问题现象**：154 个行为测试全部通过，但 Ruff 要求使用 Python 3.12 的 ``datetime.UTC``；
  mypy 认为测试中直接把 ``str`` 传给声明为 ``AnyHttpUrl`` 的 Settings 参数不够严格。
- **产生原因**：运行时 Pydantic 会把字符串转换为 URL，但静态类型检查不会假设这种隐式转换；
  同时 Ruff 的升级规则推荐更简洁的新标准库别名。
- **排查思路**：根据检查器指出的精确位置区分风格问题和类型问题，不修改业务行为。
- **解决方法**：将 ``timezone.utc`` 改为 ``UTC``；测试配置显式构造 ``AnyHttpUrl``，重跑后
  Ruff、mypy 和 pytest 全部通过。
- **学到的知识**：Pydantic 的运行时转换能力不等于静态类型兼容；测试辅助代码也应遵守与
  生产代码相同的严格类型契约。

### 问题 12：测试无法稳定导入 scripts 下的验收函数

- **问题现象**：测试最初使用 ``from scripts.check_real_market_data import ...``。mypy 报告同一
  文件同时对应 ``check_real_market_data`` 和 ``scripts.check_real_market_data``，pytest 则报告
  ``ModuleNotFoundError: No module named 'scripts'``。
- **产生原因**：项目采用 ``src`` 布局，当前 pytest 启动路径只稳定暴露安装后的 ``finagent``；
  根目录 ``scripts`` 不是正式安装包。同一文件既被 mypy 扫描又被当成包导入时产生模块歧义。
- **排查思路**：先增加 ``scripts/__init__.py`` 验证假设；这解决了 mypy 的双模块名问题，但
  pytest 仍无法保证根目录可导入，说明只补包标记并不够稳定。
- **解决方法**：删除临时 ``scripts/__init__.py``，把可测试编排移动到
  ``src/finagent/data/diagnostics.py``，脚本变成只负责启动的薄入口，测试改为导入正式模块。
- **学到的知识**：``scripts`` 适合作为人工入口，不适合作为核心可复用逻辑的唯一存放位置；
  ``src`` 布局下可测试逻辑应进入正式包，避免依赖偶然的 ``sys.path``。

### 问题 13：PR 合并后学习日记仍显示“开发中”和“PR 待创建”

- **问题现象**：PR #8 已经合并且 Issue #7 已自动关闭，但合并进入 ``main`` 的学习日记页首仍
  写着 ``v0.2.0 开发中`` 和 ``PR 待创建``。
- **产生原因**：页首元数据是在创建 PR 前编写的；PR 合并属于后续发生的外部状态，合并前的
  最后一轮文档检查只确认了实现内容，没有单独检查阶段状态字段。
- **排查思路**：合并后同时核对 GitHub PR、Issue 状态、本地 ``main`` 提交和学习日记页首，
  确认代码已经进入主分支，问题仅限文档元数据过期。首次补丁检查还发现页首沿用的 Markdown
  双空格换行会被 ``git diff --check`` 报告为尾随空格，因此一并移除。
- **解决方法**：从最新 ``main`` 创建 ``docs/phase4-closeout`` 分支，将日期范围更新到
  2026-07-17、版本改为正式 ``v0.2.0``，并记录 PR #8 已合并。
- **学到的知识**：阶段文档在 PR 创建前只能记录“待合并”，合并后还需要一次发布后检查；
  版本号、PR 编号和 Issue 状态都属于需要核对的交付元数据。

### 真实结果解释

基金净值的时间早于运行时间是正常现象：场外基金不是盘中连续成交品种，AKShare 返回的是
最近一个已经确认的交易日单位净值。GoldAPI 时间戳采用 UTC，``+00:00`` 不是数据晚了八小时；
换算到中国标准时间需要加八小时。两个来源的时间语义不同，所以统一 ``Quote`` 必须保留带时区
时间和 ``is_delayed`` 标记，不能只保留一个没有语义的价格数字。

## Git / GitHub 新知识

- Issue 和 PR 共用仓库编号，因此本阶段 Issue 是 #7。
- 从 ``main`` 创建 ``feat/7-real-market-data`` 后，未提交的 ``.gitignore`` 修改会跟随到新
  分支，不会自动写回 ``main``。
- API Key 只写入被 Git 忽略的本地 ``.env``；``.env.example`` 只能保存空占位符。
- PR #8 使用 Squash 方式合并为 ``main`` 上的单个提交 ``d31ab81``，并通过 ``Closes #7``
  自动关闭 Issue #7。
- Squash 合并会保留最终代码内容，但不会把功能分支上的五个独立提交原样接入 ``main``；因此
  后续使用普通 ``git branch -d`` 清理本地分支时可能提示“未完全合并”。
