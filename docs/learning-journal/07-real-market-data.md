# 学习日记 07：真实基金净值与国际黄金参考价

日期：2026-07-15  
版本：v0.2.0 开发中

## 本阶段目标

在现有 ``MarketDataProvider``、``MarketDataService`` 和 ``Quote`` 契约之上，接入两个
真实但用途不同的数据源：使用 AKShare 查询场外公募基金的最新已确认净值，使用 GoldAPI
查询 XAU/CNY 国际黄金参考价。所有外部数据仍要经过统一校验，且不能把参考价、昨日净值
或估算值表述成用户能够立即成交的实时价格。

## 调研后确认的真实业务口径

### 京东金融中的浙商银行财富金积存

- 第一版由用户手工录入持有克数、持仓均价、京东买入价和京东卖出价。
- 当前页面实测买入手续费为 0，卖出手续费约为卖出金额的 0.4%。
- 京东页面的持有金额和持有收益均未扣除未来卖出手续费。
- GoldAPI 的 XAU/CNY 只能用于国际市场参考和价差比较，不能代替京东卖出价计算真实到账。

### 蚂蚁财富中的东方人工智能主题混合C（017811）

- 当前持仓来自一笔买入，第一版可先实现单批次输入，但模型应为未来多批次保留扩展边界。
- 持有金额等于持有份额乘以最新已确认单位净值，不是盘中实时成交价。
- 交易日 15 点前申请赎回，最终按当天基金公司确认的净值结算；提交时只能根据上一已知净值
  给出估算金额。
- 当前页面实测持有 7～29 天的赎回费率为 0.5%，实际费用以基金公司确认结果为准。

## 数据源选型

- **基金净值主源**：AKShare，第一只验收基金为 017811。
- **基金权威核对源**：东方基金官网。
- **国际黄金参考价**：GoldAPI XAU/CNY。
- **个人持仓与京东双向报价**：第一版手工输入，后续再研究本地截图识别。

## 核心原理（用自己的话）

### 1. 可选数据源不应该阻断其他功能

百炼密钥是当前 CLI Agent 启动的必需配置；GoldAPI 只是 Phase 4 新增的可选行情源。如果在
构造任何 ``Settings`` 时都强制 GoldAPI 密钥存在，那么没有使用黄金功能的聊天、基金测试
和离线单元测试也会失败。因此配置层允许 ``goldapi_api_key=None``，只有创建 GoldAPI
Provider 时才通过 ``require_goldapi_api_key`` 明确报错。

### 2. 缓存只能保存统一模型

``QuoteCache`` 保存已经校验的 ``Quote``，不保存 GoldAPI 原始 JSON 或 AKShare DataFrame。
这样缓存命中和真实请求返回的对象遵守同一契约，上层也不会因为缓存而依赖供应商字段。

### 3. TTL 使用单调时钟

缓存关心“过去了多少秒”，不关心当前日历时间。系统时间可能因为人工修改或网络校时发生
跳变，而 ``time.monotonic`` 只向前推进，更适合判断 15 分钟是否已经过去。测试通过注入
可控时钟推进时间，不需要真正等待。

## 遇到的问题

### 问题 1：Git status 扫描 Codex 临时测试目录时提示权限不足

- **问题现象**：``git status --short`` 提示无法打开 ``.codex_tmp/pytest-of-xmz`` 和
  ``.codex_tmp/pytest_cache``。
- **产生原因**：项目根目录中的工具临时目录没有被 Git 忽略，其中部分子目录权限不允许
  当前 Git 进程遍历。
- **排查思路**：确认 ``.codex_tmp`` 没有被 Git 跟踪，并使用忽略规则阻止 Git 继续扫描。
- **解决方法**：在 ``.gitignore`` 中加入 ``.codex_tmp/``，随后 ``git status`` 不再警告。
- **学到的知识**：运行缓存和测试临时文件既不属于源码，也可能包含特殊权限，应该在创建后
  及时加入忽略规则。

### 问题 2：首次运行 uv add 无法访问用户级缓存

- **问题现象**：安装 AKShare 和 httpx 时，uv 无法打开
  ``C:\Users\xmz\AppData\Local\uv\cache``，返回 ``os error 5``。
- **产生原因**：当前受控执行环境默认只能写项目工作区，而 uv 的全局缓存位于工作区外。
- **排查思路**：错误发生在依赖解析前，路径明确指向用户缓存，因此不是包版本冲突或网络故障。
- **解决方法**：获得用户授权后允许 uv 访问其用户级缓存，再次运行安装成功。
- **学到的知识**：依赖安装同时涉及项目文件、全局缓存和网络；权限问题要与解析失败、下载失败
  分开诊断。

### 问题 3：uv 修正第三方包的非法版本声明

- **问题现象**：安装时多次出现 ``Fixing invalid version specifier``，把 ``>=3.6,`` 修正为
  ``>=3.6``。
- **产生原因**：AKShare 某个传递依赖的包元数据包含尾随逗号，不符合当前版本规范。
- **排查思路**：继续观察最终解析、构建和安装结果，而不是看到 warning 就判断安装失败。
- **解决方法**：uv 自动规范化声明，最终成功安装 AKShare 1.18.64 及其依赖。
- **学到的知识**：warning 表示需要关注但不等于失败；应以退出码、锁文件和后续导入测试作为
  最终证据。

### 问题 4：GitHub 返回的 Issue 预期目录疑似被截断

- **问题现象**：检查 Issue #7 时，正文在 ``src/finagent/data/cache.py`` 后结束，未看到测试、
  验收脚本、学习日记目录和 Markdown 代码围栏结尾。
- **产生原因**：目前无法确认是网页粘贴漏项还是连接器返回被截断。
- **排查思路**：要求在 GitHub 网页直接查看正文末尾，以网页实际内容为准。
- **当前处理**：已提供完整的“预期目录”替换内容；待后续确认网页是否已经补全。
- **下一步**：创建 PR 前再次检查 Issue 正文和任务清单是否完整。

### 问题 5：pytest 无法写入系统临时目录和旧缓存目录

- **问题现象**：首轮 pytest 中 114 项通过，但使用 ``tmp_path`` 的配置测试在创建
  ``C:\Users\xmz\AppData\Local\Temp\pytest-of-xmz`` 时触发 ``PermissionError``；同时
  ``.pytest_cache`` 出现写入警告。
- **产生原因**：受控执行环境不能写用户系统临时目录，仓库中旧 pytest 缓存也带有当前进程
  无法写入的权限状态，并非业务测试断言失败。
- **排查思路**：根据 traceback 区分“fixture 建立失败”和“被测代码失败”；全部已执行测试均
  通过，错误集中在临时目录创建阶段。
- **解决方法**：通过 ``--basetemp`` 和 ``cache_dir`` 把临时文件显式定向到已被 Git 忽略的
  ``.codex_tmp`` 子目录，重跑后 115 项全部通过。
- **学到的知识**：测试失败报告需要区分 failure 与 error；前者通常是断言不符合预期，后者
  可能发生在 fixture、收集或环境准备阶段。

### 问题 6：Git 提示 LF 将在工作区转换为 CRLF

- **问题现象**：``git diff`` 对若干文本文件提示下次写入时会把 LF 转换成 CRLF。
- **产生原因**：Windows Git 的行尾转换配置与仓库当前文件行尾不同。
- **排查思路**：``git diff --check`` 已通过，说明没有尾随空格或补丁格式错误；该信息属于
  工作区行尾提示，不是代码行为或测试失败。
- **当前处理**：本阶段不批量改写全仓库行尾，避免产生与功能无关的大面积 diff。
- **下一步**：后续生产化阶段再统一评估是否增加 ``.gitattributes`` 固定文本行尾策略。
