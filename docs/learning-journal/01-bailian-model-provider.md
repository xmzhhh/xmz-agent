# 学习日记 01：接入百炼模型 Provider

## 本阶段目标

在不让 Agent 业务代码依赖具体厂商 SDK 的前提下，完成阿里云百炼千问模型的调用层。

## 完成内容

- 定义 `BailianModelProvider`，负责统一请求与百炼请求之间的双向转换。
- 支持普通对话、Function Calling 工具定义、工具调用参数解析和 token 统计。
- 建立鉴权、限流、超时、连接、API 和响应格式六类异常边界。
- 增加可直接运行的 `scripts/check_bailian.py` 连通性检查脚本。
- 使用模拟客户端完成 5 个离线测试；项目当前共 25 个测试。

## 核心知识

### 为什么需要 Provider

如果 Agent 直接调用 OpenAI SDK，未来改用 Ollama 时，Agent 循环、消息管理和工具调用
都可能需要修改。Provider 使用统一的 `ModelRequest` 和 `ModelResponse` 隔离厂商差异，
上层只依赖稳定接口。它本质上是适配器模式。

### 单元测试和真实联调的分工

单元测试使用模拟客户端，运行快、没有费用，也不会受到网络波动影响；手动联调脚本
用于验证 API Key、网络、区域和云端模型是否真正可用。二者解决的问题不同，不能互相
替代。

### 为什么统一异常

SDK 异常属于基础设施细节。转换成项目异常后，上层可以稳定地决定是否重试、提示用户
检查密钥，或记录故障，而不用为每个模型厂商编写一套判断。

## 验证结果

- `pytest`：25 个测试通过。
- `ruff check`：通过。
- `ruff format --check`：通过。

## 本阶段问题与排查记录

### 1. PyCharm 运行时提示缺少 `llm_api_key`

- **现象**：项目根目录已经存在 `.env`，但 `Settings` 仍提示 `llm_api_key` 字段缺失。
- **原因**：原配置使用相对路径 `.env`，它会相对于进程工作目录解析；PyCharm 运行
  `scripts/check_bailian.py` 时的工作目录不一定是项目根目录。
- **排查思路**：错误发生在创建 Provider 之前，说明程序还没有发出网络请求；结合
  Pydantic 的 `input_value={}`，可以判断没有加载到任何配置。
- **解决方法**：在 `config.py` 中根据源码位置计算项目根目录，并使用绝对路径加载
  `.env`，使 PyCharm、CMD、pytest 等启动方式行为一致。
- **学到的知识**：相对路径依赖当前工作目录，而当前工作目录不一定等于脚本目录或
  项目目录；配置文件路径应使用明确、稳定的定位规则。

### 2. 真实调用出现 TLS 连接失败

- **现象**：OpenAI SDK 抛出 `APIConnectionError`，底层错误为 `httpcore.ConnectError`；
  Windows `curl` 也无法完成 TLS 握手。
- **原因**：当时所用网络环境无法与百炼域名完成 HTTPS/TLS 握手，并非 API Key、
  模型名称或 Provider 请求格式错误。
- **排查思路**：先确认 `.env` 已加载，再检查代理环境变量、WinHTTP 代理，并分别使用
  Windows `curl` 和强制 IPv4 测试，以区分 Python 代码问题与整机网络问题。
- **解决方法**：更换可正常访问百炼的网络环境后重新运行，真实调用成功。
- **学到的知识**：网络异常、鉴权异常和 API 参数异常发生在调用链的不同阶段；应根据
  是否建立连接、是否收到 HTTP 响应逐层定位，不能一看到调用失败就重新生成 API Key。

### 3. Ruff 报告导入顺序错误

- **现象**：测试通过，但 `ruff check` 报告 `I001 Import block is un-sorted`。
- **原因**：新增模块导入没有遵循 Ruff 的排序规则。
- **解决方法**：运行 `ruff check --fix .` 自动整理，并重新执行完整质量检查。
- **学到的知识**：功能测试通过不代表代码质量检查全部通过；测试、格式、Lint 和类型
  检查关注的是不同维度。

### 4. mypy 报告外部数据和 Mock 可能为空

- **现象**：mypy 指出 `finish_reason` 可能不是字符串，测试中的 `await_args` 可能为
  `None`。
- **原因**：外部 SDK 数据在静态类型层面不完全可信；Mock 只有在被调用后才一定拥有
  调用参数。
- **解决方法**：解析枚举前显式判断字符串类型；读取 Mock 参数前先断言调用确实发生。
- **学到的知识**：严格类型检查会迫使代码明确处理边界状态，这些检查也能成为运行时
  数据校验和测试意图的一部分。
