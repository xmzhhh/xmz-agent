# 学习日记 01：接入百炼模型 Provider

## 本阶段目标

在不让 Agent 业务代码依赖具体厂商 SDK 的前提下，完成阿里云百炼千问模型的调用层。

## 完成内容

- 定义 `BailianModelProvider`，负责统一请求与百炼请求之间的双向转换。
- 支持普通对话、Function Calling 工具定义、工具调用参数解析和 token 统计。
- 建立鉴权、限流、超时、连接、API 和响应格式六类异常边界。
- 增加可直接运行的 `scripts/check_bailian.py` 连通性检查脚本。
- 使用模拟客户端完成 5 个离线测试；项目当前共 25 个测试。

## 核心知识

### 为什么需要 Provider

如果 Agent 直接调用 OpenAI SDK，未来改用 Ollama 时，Agent 循环、消息管理和工具调用
都可能需要修改。Provider 使用统一的 `ModelRequest` 和 `ModelResponse` 隔离厂商差异，
上层只依赖稳定接口。它本质上是适配器模式。

### 单元测试和真实联调的分工

单元测试使用模拟客户端，运行快、没有费用，也不会受到网络波动影响；手动联调脚本
用于验证 API Key、网络、区域和云端模型是否真正可用。二者解决的问题不同，不能互相
替代。

### 为什么统一异常

SDK 异常属于基础设施细节。转换成项目异常后，上层可以稳定地决定是否重试、提示用户
检查密钥，或记录故障，而不用为每个模型厂商编写一套判断。

## 验证结果

- `pytest`：25 个测试通过。
- `ruff check`：通过。
- `ruff format --check`：通过。
- `mypy` 严格模式：通过。
- 真实调用：本机在代理/TLS 连接阶段失败，尚未收到百炼响应，因此下一步需要检查代理
  或在 PyCharm 中直接运行脚本，判断是否仅为当前执行环境的网络限制。
