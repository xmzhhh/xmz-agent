## 核心原理（用自己的话）

Provider 是“翻译官”：它知道某家供应商的 URL、字段和异常，并把外部响应转换成统一
`Quote`。Service 是“公共规则层”：无论使用哪家供应商，都要遵守相同的请求超时、代码
匹配、批量顺序和行情新鲜度规则。PortfolioCalculator 只接收合法 Quote 并计算，不关心
前两层如何取数。

这种分层避免把 HTTP、限流和供应商字段写进投资组合公式。以后切换数据源时，只新增或
替换 Provider，计算器与 Agent 不需要重写。

## 数据流

```text
资产代码列表
  → normalize_symbol
  → MarketDataService（重复检查、超时、新鲜度）
  → MarketDataProvider
  → Quote
  → PortfolioCalculator
  → PortfolioSnapshot
```

## 关键设计与取舍

### 1. Provider 接口保持最小

Provider 只声明 `get_quote(symbol)` 和 `close()`。部分供应商支持批量端点，部分只支持
单代码查询；先稳定最小共同能力，批量调度由 Service 负责。以后真实 Provider 可以在不
破坏调用方的前提下扩展优化。

### 2. 超时放在 Service

真实 SDK 可能有自己的连接超时，但应用仍需要统一的“最多愿意等待多久”。
`asyncio.timeout` 超时后会取消正在等待的协程，并转换成 `MarketDataTimeoutError`。这样
上层不需要识别 `asyncio.TimeoutError` 或不同 HTTP 客户端异常。

### 3. 批量请求暂时串行

串行调用更容易遵守免费 API 的频率限制，结果和第一个失败位置也更确定。当前演示规模
很小，并发收益有限。选定真实供应商并确认其限流规则后，再决定批量端点或受控并发。

### 4. 固定时钟通过依赖注入传入

如果测试直接读取当前系统时间，今天通过的陈旧行情测试明天就会失败。Service 接收
`clock` 函数，生产环境默认返回当前 UTC，测试使用固定时间，从而精确覆盖 30 分钟边界。

### 5. 请求代码必须与响应代码一致

请求 AAA 却收到 BBB 不是普通缺失，而是供应商响应契约错误。Service 在 Quote 进入组合
计算前比较代码并抛出 `MarketDataResponseError`，防止错误价格估值到其他资产。

## 遇到的问题

### 问题 1：指定的 grill-me 技能本轮不可用

- **问题现象**：用户要求使用 `grill-me`，但它不在当前会话的已加载技能列表中。
- **产生原因**：技能尚未安装到 Codex 用户级技能目录。
- **排查思路**：使用技能发现流程查询 skills.sh，并核对安装量、来源与安全审计。
- **解决方法**：确认技能来自 `mattpocock/skills`，安装量较高且审计通过后进行安装。
- **学到的知识**：外部技能要先验证来源和质量，安装后的技能通常从下一轮会话开始生效。

### 问题 2：全局 python 命令无法运行安装器

- **问题现象**：首次运行技能安装脚本退出代码为 1 且没有错误文本；进一步检查发现
  Windows 的全局 `python.exe` 无法访问，`py` 命令也不存在。
- **产生原因**：本机 Python 由 uv 项目环境提供，没有可用的全局 Python 启动器。
- **排查思路**：分别检查 `python --version`、`py --version` 和项目 `.venv` 解释器。
- **解决方法**：使用 `C:\Users\xmz\Desktop\xmz-agent\.venv\Scripts\python.exe`
  运行安装脚本。
- **学到的知识**：Windows 命令解析到的 python 不一定是项目解释器；诊断时要确认实际
  可执行文件路径。

### 问题 3：grill-me 在仓库中的路径与预期不同

- **问题现象**：使用 `skills/grill-me` 安装时提示 Skill path not found。
- **产生原因**：该仓库按类别组织技能，真实路径是 `skills/productivity/grill-me`。
- **排查思路**：通过 GitHub 搜索并读取仓库的 productivity README，确认链接目标。
- **解决方法**：使用正确子目录重新安装，最终成功写入用户级技能目录。
- **学到的知识**：包名和仓库目录不一定相同，应读取仓库索引而不是猜测路径。

### 问题 4：实时年龄测试不能依赖系统当前时间

- **问题现象**：需要验证“30 分钟有效、31 分钟陈旧”，若直接调用当前时间，测试结果会
  随运行时刻变化。
- **产生原因**：时间属于隐式外部依赖。
- **排查思路**：把当前时间从隐藏的系统调用变成构造函数参数。
- **解决方法**：注入 `clock=lambda: FIXED_NOW`，稳定验证边界。
- **学到的知识**：时间、随机数、网络和文件系统都适合通过依赖注入提升可测试性。

### 问题 5：新增文件首次需要 Ruff 格式化

- **问题现象**：两个新 Python 文件首次检查时需要统一格式化。
- **产生原因**：长参数列表和测试数据在初稿后尚未经过 Ruff 排版。
- **排查思路**：先执行 format，再分别运行 lint、mypy 和 pytest。
- **解决方法**：自动格式化后第一轮静态类型和运行测试全部通过。
- **学到的知识**：格式通过只代表排版一致，仍需类型检查和行为测试提供不同证据。

## 仍然不懂的地方

- 真实数据源的授权、商用许可和展示限制如何评估。
- 连接超时、读取超时和应用总超时如何组合。
- 免费 API 的限流如何实现 token bucket 或指数退避。
- 缓存命中时如何同时表达缓存年龄和原始行情时间。
- 多数据源价格冲突时如何选择主源和降级源。
