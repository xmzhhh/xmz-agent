## 本阶段目标

把模糊的“AI 投资管家”收敛为可开发、可评测、可写进简历的 Agent 项目，并设计渐进式路线。

## 我们确定了什么

- 项目定位为“可追溯的 AI 投资研究与资产监控助手”。
- 第一版只使用模拟持仓，不接真实交易。
- 数值由确定性程序计算，LLM 负责理解、规划和报告生成。
- 每条事实必须有来源和时间；每个高风险建议必须经人工确认。
- 先完成单 Agent 和基础工具，再学习记忆、RAG、工作流、本地模型、MCP 与多 Agent。
- 使用模块化单体和抽象接口，保证模型、数据源和存储可以替换。

## 关键设计取舍

### 为什么不直接做自动交易

自动交易引入资金安全、合规、实时行情和券商接口等巨大复杂度，却不能直接证明 Agent 工程能力。先做好研究、监控、证据链和人工确认，更安全，也更适合求职展示。

### 为什么不一开始上多智能体

多智能体会增加成本、延迟、共享状态和调试难度。没有单 Agent 基线，就无法证明它是否真的改善效果。因此第 8 阶段才通过评测决定是否保留多 Agent。

### 为什么先手写最小循环再用框架

手写一次模型与工具的调用闭环，可以理解消息、Schema、终止条件和失败处理；之后使用工作流框架时，才能知道框架替我们解决了什么。

## 已发现的风险

- 不同市场的数据授权、频率和字段不同，需要数据适配层。
- 金融新闻具有时效性和观点偏差，必须保存来源与有效时间。
- 本地模型可能不稳定支持工具调用和结构化输出，需要能力探测和降级。
- RAG “能返回内容”不等于回答可信，必须建立检索和引用评测。
