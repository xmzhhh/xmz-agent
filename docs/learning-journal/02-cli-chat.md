# 学习日记 02：实现 `finagent chat` 多轮对话

日期：2026-07-03  
版本：v0.1.0  
相关 Issue / PR：Issue #1 / PR #2

## 本阶段目标

把已经跑通的百炼 Provider 接入正式 CLI，完成“用户输入—模型回答—保留上下文—继续
追问”的第一个可交互闭环。

## 我实现了什么

- 新增 `ChatSession`，在内存中维护 system、user、assistant 多轮消息。
- 新增 `finagent chat` 子命令和终端输入循环。
- 支持 `exit`、`quit`、`退出`、Ctrl+C 和终端 EOF 等退出方式。
- 拦截空输入，避免无意义模型调用和 token 消耗。
- 将模型异常转换为用户可理解的中文提示。
- 通过依赖注入使用假 Provider、输入函数和输出函数完成离线测试。

## 核心原理（用自己的话）

CLI 是用户与程序交互的入口，不应该同时承担对话状态和模型协议转换。`cli.py` 只处理
命令、输入和输出；`ChatSession` 管理对话历史；`BailianModelProvider` 管理百炼接口。
这种分层使未来的 Web API 可以直接复用会话层，也使本地模型能够替换百炼 Provider。

多轮对话并不是模型自动记住了用户，而是应用在每次请求时把之前的消息重新发送给
模型。当前实现将历史保存在内存中，退出程序后会消失。

## 关键设计与取舍

- 只有模型成功返回文本后才把本轮问答写入历史，防止网络失败留下孤立的 user 消息。
- 返回工具调用但没有文本时暂时报错，因为本阶段尚未实现 Agent 工具执行循环。
- 当前使用同步 `input()` 配合异步模型调用，足以满足单用户 CLI；并发界面将在 Web API
  阶段选择更合适的输入机制。
- system prompt 明确投资研究助手不承诺收益，也不替用户做最终投资决策。

## 遇到的问题

### 问题 1：Ruff 报告测试代码超过行宽

- **现象**：31 个 pytest 测试全部通过，但 Ruff 报告 `E501 Line too long`，中文 system
  prompt 的断言行达到 126 个字符。
- **假设**：功能没有错误，问题来自测试重复书写了较长的常量文本。
- **排查证据**：Ruff 精确指向 `tests/test_cli.py` 中的预期 system prompt 字符串。
- **根因**：测试复制生产常量，既超过 100 字符行宽，也增加了提示词修改后测试漏改的
  风险。
- **修复**：测试直接导入并使用 `DEFAULT_SYSTEM_PROMPT`，消除重复字符串。
- **如何防止复发**：稳定且需要精确一致的公共常量应集中定义；完成 pytest 后仍要继续
  执行 Ruff、格式和 mypy，不能把“测试通过”等同于全部检查通过。
